# DGentic

DGentic is an advanced autonomous AI agent platform concept focused on local and external model orchestration, dynamic sub-agent spawning, backend-managed agent task graphs, guarded system access, persistent memory, reusable tools, and developer-facing interfaces.

The current repository contains the project specification, planning documents, and a backend MVP surface for orchestrator planning, deterministic execution runs, backend-managed orchestration runs with detached background execution polling and cooperative cancellation, production/staging bearer-token capability gates with startup fail-closed validation, persisted operator/group identity records, and persisted hashed token lifecycle APIs, guardrail checks, guarded filesystem operations with single-use bound approval records, guarded CLI execution with single-use bound approval IDs, guarded git workflow checkpoints, checkpoint-bound git commit/push/PR approval creation, direct checkpoint-bound local git commit/configured-upstream push/GitHub PR creation execution, managed policy surface locks, deployment-managed command recipes, deployment-managed inert plugin component records, cwd-aware rootDir command policy checks, read-only path operand, explicit executable path, and configured-safe command path-argument hardening, bare executable workspace/PATH trust checks, top-level `cmd` AutoRun and PowerShell profile suppression, asynchronous CLI runs with status polling, chunked output polling, supervision metadata, auditable lifecycle states, cancellation, and stale-running reconciliation, configurable command policy rules with agent-role scoping, backend hook-policy records for command/filesystem/network pre-action escalation, controlled and audited command environment overrides with startup/preload injection blocking, local provider probes and generation calls, network/domain policy checks for provider egress, guarded web retrieval preflights and bounded text fetch runtime, generated-tool Python socket egress guardrails with single-use host/port network approval consumption, scored provider routing, agent lifecycle tracking, memory records, dynamically generated and executable local tools, backend-only plugin manifest discovery/trust records with declarative command recipe activation, tool governance, session summaries, event logs, local JSON state persistence, and a migration-managed SQLAlchemy persistence baseline with SQLite backup/restore helpers for metadata and tool registry tables.

## Quick Start

Install dependencies:

```powershell
uv sync --dev
```

Run the backend:

```powershell
uv run uvicorn dgentic.main:app --reload --app-dir src
```

Or use the installed command after package installation:

```powershell
dgentic-server --host 127.0.0.1 --port 8000
```

Run tests:

```powershell
uv run pytest
```

Run all quality checks:

```powershell
uv run pytest
uv run ruff check .
uv run ruff format --check .
```

Authentication is off by default in `development`, on by default in `staging` and `production`, and can be explicitly controlled with `DGENTIC_AUTH_ENABLED`. Protected routes can use bootstrap bearer tokens from `DGENTIC_AUTH_TOKENS`, for example `admin-token=admin;task-token=tasks`, or persisted generated tokens stored as salted PBKDF2 hashes in `auth-tokens.json` and assigned to persisted operator profiles in `operators.json`. Operator groups live in `operator-groups.json`; operators can reference them through `group_ids`, and token issuance plus runtime authorization are constrained by the operator's current direct plus active group-inherited effective capabilities. Operator display/role metadata, operator group display/description metadata, generated-token labels, credential-reference labels, and plugin trust reasons are redacted for common secret-shaped values before API responses, audit metadata, and new or mutated JSON persistence. Capability groups include `auth`, `credentials`, `tasks`, `filesystem`, `cli`, `providers`, `approvals`, `network`, `hooks`, `agents`, `memory`, `tools`, `sessions`, `logs`, and `admin`. Plugin discovery and trust routes use the `tools` capability; plugin command recipe activation routes require `tools` plus `cli` or `admin`, and plugin hook-policy activation routes require `tools` plus `hooks` or `admin` when auth is enabled. Hook policy rule management uses the `hooks` capability. The effective-settings view is admin-only when authentication is enabled. Filesystem approval creation and bound filesystem execution use the `filesystem` capability, while filesystem approval list, review, approve, and deny routes use the separate `approvals` capability. CLI approval creation and approved-command execution use the `cli` capability, while CLI approval list, review, approve, and deny routes use the separate `approvals` capability. When authentication is enabled, application startup fails closed if no usable environment token or active persisted token is configured.

Set `DGENTIC_MANAGED_SETTINGS_FILE` to opt into an organization-managed JSON settings file. The file must contain a top-level `settings` object, for example `{"settings":{"auth_enabled":true,"network_domain_policy":{"default_mode":"deny","rules":[]},"managed_policy_locks":["cli_policy"]}}`. Managed values override `.env` and process environment values, unknown or malformed fields fail closed during settings load, and `GET /settings/effective` reports redacted values with `default`, `environment`, `managed`, or `derived` source labels plus a SHA-256 digest for the managed file. `managed_policy_locks` can make supported mutable policy surfaces read-only to API mutation while preserving read/preview/evaluation access. The managed file can also declare read-only `managed_credential_references`, `credential_secret_manager_adapters`, `credential_secret_manager_allowed_base_urls`, `managed_cli_policy_rules`, `managed_hook_policy_rules`, `managed_network_domain_policy_rules`, `managed_command_recipes`, `managed_plugin_component_records`, and `managed_plugin_trust_records`; managed credential references, secret-manager adapter settings, CLI/hook/network rules, command recipes, and plugin component records are source-attributed as `managed`, returned by their list/detail routes where applicable, excluded from local mutable JSON persistence, and rejected by normal mutation routes, while managed plugin trust records are exact-manifest-digest scoped and make local trust mutation for those plugin ids read-only.

Local state is stored under `.dgentic/` by default and is ignored by Git. Override JSON state with `DGENTIC_DATA_DIR` and SQLAlchemy state with `DGENTIC_DATABASE_URL` when needed. Malformed or schema-invalid JSON collection files are quarantined in-place and the active collection is repaired to an empty valid file so the backend can continue starting. File-backed SQLite databases can be backed up and restored with the `backup_sqlite_database` and `restore_sqlite_database` helpers for local/operator smoke workflows.

### Runtime Usage

Current backend MVP usage is through the HTTP API. The human-facing chat/web UI is planned for Sprint 16, and the dedicated CLI plus VS Code extension are planned for Sprint 17. As those interfaces land, DGentic should be used through one or more supported interfaces:

1. Start the DGentic backend orchestrator.
2. Configure local model runtimes such as Ollama or LM Studio.
3. Add external provider credentials through the secure settings layer.
4. Set the workspace `rootDir` for guarded file and CLI access.
5. Submit a task through the current API, then through the chat interface, dedicated CLI, or VS Code extension as those clients are implemented.
6. Review the orchestrator plan, approve restricted actions, and inspect sub-agent progress.
7. Export or persist the session summary, created tools, and memory updates.
